| Issue                                   | Atributo de Calidad | Riesgo/falla                                                                 | ¿Por qué?                                                                 | Respuesta del equipo |
|----------------------------------------|---------------------|------------------------------------------------------------------------------|---------------------------------------------------------------------------|----------------------|
| Caídas del sistema 24/7                | Disponibilidad      | Dependencia de múltiples componentes críticos (dispositivo, conexión y aplicación) sin mecanismos de redundancia. | Fallas en servidores, conexión o batería del dispositivo.                 | Aceptado. La disponibilidad continua es un requisito importante del sistema. Las fallas de conexión, batería o servidor afectan directamente el funcionamiento principal, por lo que es un riesgo crítico que debe mitigarse. |
| Fallas en actualización de la base de datos | Confiabilidad       | Si ocurre una falla en la conexión o en la base de datos, la información no se guardará correctamente, generando pérdida o incoherencias en los datos. | Problemas de conexión, errores en el motor de base de datos o falta de sincronización. | Aceptado. La pérdida o inconsistencia de datos compromete la confiabilidad del sistema. Es un riesgo real en sistemas distribuidos con conexión intermitente, por lo que se debe manejar con mecanismos de sincronización y recuperación. |
| Bluetooth como único canal de comunicación | Disponibilidad      | Interrupción del envío de datos en tiempo real entre el collar y la aplicación. | Dependencia de un único canal sin alternativas como Wi-Fi o 4G.           | Aceptado. Es un riesgo arquitectónico claro, ya que toda la comunicación depende de un único canal. Si Bluetooth falla, el sistema deja de funcionar, lo cual impacta directamente la disponibilidad. |
| Falla de sensores                      | Rendimiento         | El diseño actual no define qué debe hacer el sistema cuando un sensor devuelve valores fuera de rango o se desconecta. | No existe manejo de excepciones ni control de errores para los sensores.   | Aceptado. Los sensores son la base del sistema. No manejar errores o valores inválidos puede generar datos incorrectos o pérdida de funcionalidad, por lo que debe considerarse en el diseño. |
| Modelo de IA sin precisión             | Confiabilidad       | El sistema puede mostrar al usuario una emoción incorrecta como si fuera completamente certera, sin indicar el nivel de confianza. | Falta de entrenamiento adecuado del modelo, validación insuficiente o ausencia de fallback. | Aceptado. La precisión del modelo impacta directamente la confiabilidad del sistema. Mostrar resultados incorrectos sin indicar incertidumbre puede afectar la experiencia del usuario y la credibilidad del sistema. |
